# ToolContext Guide

`ToolContext` declares a host-owned value that a tool needs but the model must
not generate. It is annotation syntax only: the parameter is hidden from the
model-facing input schema, metadata parameter list, and public tool event
input.

## Required Context

Use `ToolContext[T]` for a required value that the host supplies for one tool
invocation. The payload may be a normal class or a dataclass.

```python
from dataclasses import dataclass

from easyharness import Agent, ModelConfig, ToolContext, tool


@dataclass(frozen=True)
class RequestContext:
    request_id: str
    user_id: str


@tool(
    name="read_profile",
    purpose="Read the current user profile.",
    when_to_use="Use when the user asks for their profile.",
    parameters=dict(),
    returns="A profile summary.",
    common_failures="The current request is unavailable.",
)
def read_profile(request: ToolContext[RequestContext]) -> str:
    return f"profile for {request.user_id}"


agent = Agent(
    model=ModelConfig(model="example", api_key="example"),
    system_prompt="Answer profile questions.",
    tools=[read_profile],
)
result = agent.run(
    "Show my profile.",
    request=RequestContext(request_id="req-42", user_id="user-7"),
)
```

The model cannot construct or supply `request`; the host supplies it by its
parameter name for the current `run()` or `stream()` call.

## Optional Context

Use `OptionalToolContext[T]` when an omitted host value is valid and should
resolve to `None`.

```python
from easyharness import OptionalToolContext, tool


@tool(
    name="read_profile_summary",
    purpose="Read a profile when request data is available.",
    when_to_use="Use when the request may be anonymous.",
    parameters=dict(),
    returns="A profile summary or an anonymous marker.",
    common_failures="The profile lookup fails.",
)
def read_profile_summary(
    request: OptionalToolContext[RequestContext],
) -> str:
    if request is None:
        return "anonymous request"
    return f"profile for {request.user_id}"


assert read_profile_summary() == "anonymous request"
```

The decorated callable exposes `request=None` in its inspectable signature
when this optional parameter has no source default. This means callers may omit
an optional Context after ordinary arguments:

```python
def render_message(
    text: str,
    request: OptionalToolContext[RequestContext],
) -> str:
    return text if request is None else f"{text}:{request.request_id}"


assert render_message("hello") == "hello"
```

## Default Resolution Order

Direct calls and Agent tool invocations use the same resolution order for each
Context parameter:

1. Use an explicit host value supplied by the Context parameter name.
2. Otherwise use a default written on the original function parameter.
3. Otherwise inject `None` for `OptionalToolContext[T]`.
4. Otherwise fail safely before the tool body runs.

A declared default is preserved for both required and optional Context:

```python
DEFAULT_REQUEST = RequestContext(request_id="default", user_id="system")


@tool(
    name="read_default_profile",
    purpose="Read a profile using a host default when necessary.",
    when_to_use="Use only when the default profile is allowed.",
    parameters=dict(),
    returns="A profile summary.",
    common_failures="The request Context is invalid.",
)
def read_default_profile(
    request: ToolContext[RequestContext] = DEFAULT_REQUEST,
) -> str:
    return request.user_id


assert read_default_profile() == "system"
```

## Supported Payload Types

A Context payload must be one concrete, runtime-checkable type expression. The
following are supported recursively:

- Normal classes, including subclasses of the declared parent class.
- `object`.
- `dict[K, V]`, `list[T]`, `set[T]`, `tuple[T, ...]`, and fixed tuples such as
  `tuple[str, int]`.

```python
@tool(
    name="inspect_context_payload",
    purpose="Inspect host-owned structured data.",
    when_to_use="Use only when the host supplies validated structured data.",
    parameters=dict(),
    returns="A fixed status marker.",
    common_failures="The Context payload is invalid.",
)
def inspect_context_payload(
    records: ToolContext[dict[str, list[RequestContext]]],
    marker: ToolContext[tuple[str, int]],
    labels: OptionalToolContext[set[str]],
) -> str:
    del records, marker, labels
    return "accepted"
```

There is no coercion. For example, a dictionary cannot stand in for a declared
`RequestContext` object.

## Declaration Restrictions

Do not use payload unions. They fail during `@tool` registration, including
unions nested inside supported containers:

```python
request: ToolContext[RequestContext | AdminContext]
request: ToolContext[dict[str, RequestContext | AdminContext]]
request: ToolContext[RequestContext] | AdminContext
```

Do not use `Any`, `None`, `Protocol`, unsupported `typing` forms, positional-only
parameters, `*args`, or `**kwargs` in a tool Context contract. A Context
parameter must follow all ordinary positional-or-keyword parameters.

`ToolContext` and `OptionalToolContext` are not base classes and must not be
subclassed. They are annotation helpers, not domain model inheritance points.

## Compatibility and Migration

`ToolContext[T] | None` remains accepted for compatibility, but it emits a
`DeprecationWarning`. Use `OptionalToolContext[T]` in new code.

`ToolContext[object]` accepts an explicitly supplied `None` for compatibility,
but emits a `RuntimeWarning` recommending `OptionalToolContext[object]` when
that value is used. Prefer the optional annotation whenever `None` is part of
the intended contract.

## Shared Agent Contracts

Multiple tools may use the same Context parameter name when their normalized
payload annotations are identical. Required and optional declarations may be
mixed for that shared name. Different payload annotations for the same name are
rejected when the Agent is created.

## Failure, Scope, and Privacy

A missing required Context or a wrong payload type prevents the tool body from
running. Error messages identify the parameter and expected type without
rendering the supplied Context value. Unknown Context names are rejected before
Agent invocation begins.

Context values live only for one `run()` or `stream()` invocation. EasyHarness
does not construct, serialize, cache, or reuse them. Tool output remains normal
tool output: if a tool deliberately returns Context data, that returned data may
be visible to the model.

## Checklist

1. Use `ToolContext[T]` for host-owned values hidden from the model.
2. Use `OptionalToolContext[T]` when omission should resolve to `None`.
3. Keep Context names out of `parameters` metadata.
4. Keep Context parameters after ordinary positional-or-keyword parameters.
5. Use one concrete, supported payload type with no unions.
6. Pass host values by the exact Context parameter name when a tool needs them.
7. Treat tool output as model-visible unless the tool explicitly redacts it.
