# ToolContext Guide

This guide describes the current EasyHarness Context contract. A Context is a
host-owned dependency for a tool invocation. It is not a model input, global
state container, or permission system.

## Required Context

Use `ToolContext[PayloadType]` when a tool needs one required host-provided
value. The payload type is an ordinary application type and must not inherit
from an EasyHarness base class.

```python
from dataclasses import dataclass

from easyharness import Agent, ModelConfig, ToolContext, tool


@dataclass(frozen=True)
class RequestContext:
    request_id: str
    user_id: str


@tool(
    name="read_profile",
    purpose="Read the current user profile.",
    when_to_use="Use when the user asks for their profile.",
    parameters=dict(),
    returns="A profile summary.",
    common_failures="The current request is unavailable.",
)
def read_profile(request: ToolContext[RequestContext]) -> str:
    return f"profile for {request.user_id}"
```

The model cannot create `request`. The host supplies it by parameter name for
one agent turn:

```python
agent = Agent(
    model=ModelConfig(model="example", api_key="example"),
    system_prompt="Answer profile questions.",
    tools=[read_profile],
)
result = agent.run(
    "Show my profile.",
    request=RequestContext(request_id="req-42", user_id="user-7"),
)
```

## Optional Context

Use `OptionalToolContext[PayloadType]` when a missing host value must resolve
to `None`.

```python
from easyharness import OptionalToolContext, tool


@tool(
    name="read_profile_summary",
    purpose="Read a profile when request data is available.",
    when_to_use="Use when the request may be anonymous.",
    parameters=dict(),
    returns="A profile summary or an anonymous marker.",
    common_failures="The profile lookup fails.",
)
def read_profile_summary(
    request: OptionalToolContext[RequestContext],
) -> str:
    if request is None:
        return "anonymous request"
    return f"profile for {request.user_id}"
```

EasyHarness injects `None` when the host omits this Context during `Agent.run()`
or `Agent.stream()`. The same rule applies to direct calls of the decorated
tool:

```python
assert read_profile_summary() == "anonymous request"
```

The decorated tool signature also exposes a `None` default for an optional
Context, so inspection and direct-call behavior agree.

## Schema Boundary

Context parameters are excluded from tool metadata and model-facing JSON
schema. Only ordinary model inputs appear in `parameters` metadata.

```python
@tool(
    name="search_orders",
    purpose="Search orders by status.",
    when_to_use="Use when the user asks about orders.",
    parameters=dict(status="The order status to search for."),
    returns="Order summaries.",
    common_failures="The order service is unavailable.",
)
def search_orders(
    status: str,
    request: ToolContext[RequestContext],
) -> str:
    return f"{status}:{request.request_id}"
```

`status` is visible to the model. `request` is not visible in the schema,
metadata description, or default tool event input.

## Valid and Invalid Declarations

A Context parameter must resolve to exactly one concrete runtime-checkable
payload type.

```python
request: ToolContext[RequestContext]
request: OptionalToolContext[RequestContext]
```

The following forms fail while `@tool` registers the function:

```python
request: ToolContext[RequestContext | AdminContext]
request: ToolContext[RequestContext] | AdminContext
request: OptionalToolContext[RequestContext | AdminContext]
```

Use one shared base class or a runtime-checkable protocol if several host
objects intentionally satisfy the same Context contract. Do not use a union to
model multiple Context payload alternatives.

`ToolContext[RequestContext] | None` remains accepted only as deprecated syntax
sugar. Prefer `OptionalToolContext[RequestContext]` for every new tool.

## Removed Inheritance Contract

Do not subclass `ToolContext`:

```python
class RequestContext(ToolContext):
    pass
```

This is intentionally unsupported. `ToolContext` is annotation syntax, not a
business-domain base class.

## Invocation Failures

Required Context failures prevent the tool body from running. Wrong Context
types also fail without rendering the supplied Context value in the error text.
Unknown Context parameter names fail before agent invocation begins.

```python
agent.run("Read profile", unknown=RequestContext("req-1", "user-1"))
```

The call above fails because `unknown` is not a Context parameter registered by
any available tool.

## Lifecycle and Privacy

Context values exist only for the current `run()` or `stream()` invocation.
EasyHarness does not serialize, cache, reuse, or construct Context instances.
A second turn must receive its own host values if its tools need them.

Tool output is different: if a tool deliberately returns data from a Context,
that data can be exposed to the model. Context injection protects the input
path; it does not automatically redact the tool's own output.

## Checklist

1. Use `ToolContext[T]` only for host-owned values that the model must not
   generate.
2. Use `OptionalToolContext[T]` only when omitted host injection should become
   `None`.
3. Keep Context names out of `parameters` metadata.
4. Use one runtime-checkable payload type per Context parameter.
5. Pass host values by the exact Context parameter name on every required turn.
6. Treat Context output as normal tool output and redact it explicitly when
   necessary.
