## Context

`EasyHarness` 刚完成第一版 SDK 落地，代码规模还小，但已经暴露出三类常见后期会持续放大的质量问题：

- 结构性噪声：未使用 import、未使用形参、导入顺序不一致、长行和格式漂移；
- 类型与编辑器噪声：过宽的 `Any`、缺失注解、被 `type: ignore` 压下去的类型不匹配，以及覆盖父类接口时签名不够精确；
- 语言策略分歧：仓库基础规则要求 Python 文件顶部说明、docstring 与关键注释默认使用中文，但实现者很容易把“cleaner”误解成“全部英文”。

这不是单点修 bug，而是一次跨越 `easyharness/`、`tests/`、文档与静态检查门禁的横切整理。设计重点不是“如何批量格式化”，而是定义一套可长期维持的判断标准，避免清理完一轮后很快再次回到同样的噪声状态。

## Goals / Non-Goals

**Goals:**

- 建立一套可执行的 EasyHarness 代码洁净化标准，覆盖未使用符号、格式化、静态方法判定、类型收紧和编辑器 warning 收敛。
- 让 `easyharness/` 核心代码达到稳定且可解释的 `ruff` 基线，并把超长行、导入排序和显然无用的符号清掉。
- 明确语言策略：默认继续遵守仓库中文规则，只在外部协议、标准接口和模型可见文本等场景保留英文。
- 给后续实现一个“何时必须补注解、何时允许保留 `Any`、何时允许保留 warning”的判断框架。

**Non-Goals:**

- 不借清理名义修改 SDK 对外行为、公开 API 或运行时特性。
- 不把所有 Python 文档、注释和字符串无差别改成英文。
- 不为了追求“零 warning”而引入过度复杂的类型包装、协议层或辅助抽象。
- 不新增运行时依赖，也不把一次 clean-up 扩展成大规模重构。

## Decisions

### 1. 以“噪声分层”而不是“一把梭格式化”推进

**Decision**

把本次清理拆成四层：

1. 确定性噪声：未使用 import、未使用形参、导入顺序、可自动格式化的长行；
2. 低争议结构优化：明确不依赖实例状态的私有 helper 静态化；
3. 类型精度提升：去掉无依据的 `type: ignore`，收紧可直接表达的注解；
4. 语言与 warning 策略：明确哪些中文应保留，哪些 warning 可以接受，哪些必须消除。

**Rationale**

这样可以保证每一层都有清晰的收益和边界，不会把“清噪声”演变成“顺手改设计”。第一层优先级最高，因为收益最大且最安全；第四层最需要显式规则，否则很容易各自理解。

**Alternatives considered**

- 一次性跑 formatter/linter 后全量修：快，但会把设计判断和机械修复混在一起。
- 只删 unused import：太保守，不能解决当前反复出现的风格和类型噪声。

### 2. 语言策略以仓库规则为准，而不是以个人习惯为准

**Decision**

Python 文件顶部说明、docstring、关键注释、内部验证错误提示默认继续使用中文；以下场景允许或要求英文：

- 面向底层模型的工具描述模板；
- 外部协议字段、标准 API 参数和值；
- 与第三方框架/错误语义直接对齐的文案；
- 测试中明确要验证的英文输出。

**Rationale**

当前仓库基础规则已经把中文设为默认语言，清理动作不应越权改变这套协作规范。真正需要的是把“中文默认 + 英文例外”的边界写清楚，而不是让每次 review 都重新争论。

**Alternatives considered**

- 全量切英文：和现有仓库规则冲突，也会让已写好的中文文档策略失效。
- 完全自由混用：短期省事，长期 review 成本最高。

### 3. 静态方法只在不破坏协议/继承语义时采用

**Decision**

只有在以下条件同时满足时，方法才应改成 `@staticmethod`：

- 不使用 `self` / `cls`；
- 不参与抽象基类、`Protocol` 或 override 语义；
- 改为静态方法后不会削弱可读性或误导调用方式。

对测试辅助方法也适用同样标准，但不会机械化统一处理。

**Rationale**

“不使用 self”只是候选条件，不是充分条件。像 `Protocol` 方法、抽象接口实现、property 和 override 方法即使当前不读实例状态，也仍然是实例协议的一部分。

**Alternatives considered**

- 扫到不用 `self` 就全部静态化：最容易产生假阳性。
- 一律不动：会保留可以低风险净化的明显噪声。

### 4. 类型精度以“减少噪声并保持直接表达”为准

**Decision**

类型清理遵循以下优先级：

- 优先移除明显多余的 `Any`；
- 优先补齐公共接口、override 方法和核心内部 helper 的缺失注解；
- 优先消除无依据的 `type: ignore`；
- 对动态边界、第三方弱类型接口和测试夹具，允许保留少量 `Any`，但必须是可解释的。

测试代码不追求和生产代码同等严格，但不应长期停留在“几乎无注解”的状态。

**Rationale**

目标是让编辑器和静态检查器发出的 warning 更有信号，而不是为了零告警引入复杂类型体操。对于 SDK 这种小代码面项目，最合理的远见是“足够严格但不抽象过头”。

**Alternatives considered**

- 完全类型洁癖：会在测试和第三方边界上引入大量低收益复杂度。
- 完全接受 `Any`：会让静态检查失去价值。

### 5. 质量门禁以 `ruff` 为主，类型检查为补充预留

**Decision**

本次收敛先以仓库现有可用工具为主：

- `ruff check` 负责 unused symbols、导入排序、PEP 8 基线；
- `ruff format --check` 负责格式一致性；
- 类型策略先通过代码调整与局部注解收敛实现，后续如仓库接受，再考虑引入 `pyright` 或 `mypy` 作为补充门禁。

**Rationale**

当前环境里 `ruff` 已可直接使用，而 `mypy/pyright` 尚未安装。最合理的方案是先用现有工具完成高收益清理，再决定是否正式引入新的类型门禁，而不是为了“完美”先加一堆新约束。

**Alternatives considered**

- 立即引入完整类型检查链路：方向正确，但不应成为本次 clean-up 的前置阻塞。
- 完全不设门禁：问题会很快反弹。

## Risks / Trade-offs

- **[风险]** 清理动作可能顺手演变成结构重写  
  **Mitigation:** 严格按“确定性噪声优先、行为不变”推进；涉及行为或公开接口变化的改动不纳入本次范围。

- **[风险]** 语言策略被误读成“禁止英文”  
  **Mitigation:** 在规范里明确列出英文例外场景，尤其是模型可见描述和协议字段。

- **[风险]** 过度收紧类型导致代码可读性下降  
  **Mitigation:** 以减少 warning 为目标，而不是追求类型体操；对第三方动态边界允许保留少量 `Any`。

- **[风险]** 测试代码清理投入过高、收益过低  
  **Mitigation:** 测试代码采用“补关键缺失注解 + 清明显 warning”的中等标准，不追求和生产代码完全一致。

## Migration Plan

1. 先落语言策略与洁净化边界，明确哪些问题必须修、哪些不修。
2. 再执行确定性清理：unused import/形参、导入排序、格式化、长行。
3. 再做低风险类型和静态方法收敛，优先处理核心 SDK 实现文件。
4. 最后复跑 `ruff` 和现有测试，确认行为未变且 warning 基线下降。
5. 若后续需要更强类型门禁，再单独提出变更引入 `pyright`/`mypy`。

- 是否要在 `pyproject.toml` 中正式固化 `ruff` 规则，还是先遵循默认基线完成一次清理？
