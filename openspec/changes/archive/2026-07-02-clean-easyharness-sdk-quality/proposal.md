## Why

`EasyHarness` 已经完成第一版 SDK 公开表面，但当前代码质量状态仍然混杂着几类“低价值噪声”：未使用 import 与形参、格式与长行问题、过宽的类型注解、被忽略的类型不匹配，以及没有被明确写下的语言策略。这些问题短期不一定阻断功能，但会持续制造 review 成本、编辑器黄线和后续维护分歧，因此应该在代码面还小的时候一次性建立清晰的清理标准与门禁。

这次变更之所以现在做，是因为仓库规则已经明确要求 Python 文档与注释默认使用中文，但当前关于“哪些字符串必须英文、哪些应继续中文”的边界还没有被正式沉淀。继续靠个人直觉清理，只会把“消噪”变成新的噪声来源。

## What Changes

- 新增一套面向 `easyharness/` 与 `tests/` 的代码洁净化方案，明确哪些问题属于必须清理的噪声，哪些属于保留的设计取舍。
- 新增代码质量收敛要求：移除未使用 import/形参、修复 import 排序、统一格式化与长行策略，确保核心代码通过约定的 `ruff` 基线。
- 新增类型精度与编辑器诊断要求：减少无必要的 `Any`、移除无依据的 `type: ignore`、补齐关键缺失注解，目标是让 PyCharm/静态检查器的 warning 降到“有意保留且可解释”的最小集合。
- 新增静态方法判定规则：仅把确实不依赖实例状态且不会破坏继承/协议语义的方法改成 `@staticmethod`，禁止机械化静态化。
- 新增文档与字符串语言策略：明确 Python 文件顶部说明、docstring、关键注释默认继续使用中文；仅对面向模型协议、外部接口、标准错误文案或兼容性要求明确需要英文的字符串保留英文。
- **BREAKING** 不把“全部改成英文”定义为本次目标；若要改变仓库默认语言策略，必须先修改基础规则而不是借清理名义隐式改写。

## Capabilities

### New Capabilities
- `sdk-code-hygiene`: 定义 EasyHarness 代码中的未使用符号清理、PEP 8/ruff 基线、静态方法判定和编辑器告警收敛要求。
- `python-language-policy`: 定义 EasyHarness Python 代码中的 docstring、注释、内部错误消息与协议字符串的语言边界，避免清理动作与仓库规则冲突。
- `type-annotation-discipline`: 定义 EasyHarness 代码中 `Any`、缺失注解、`type: ignore` 与签名精度的约束，目标是建立可解释的类型基线。

### Modified Capabilities

- None.

## Impact

- 受影响目录主要是 `easyharness/`、`tests/` 与 `README.md`，以及这次新增的 OpenSpec 规范文档。
- 对外 SDK 行为不应发生功能性变化，影响重点在实现质量、代码风格、静态检查体验和文档一致性。
- 后续实现可能引入或强化 `ruff`/格式化/类型检查门禁，但不应新增运行时依赖或扩大公开 API 表面。
